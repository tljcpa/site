AI 论文排版助手

个人

写本科毕设时被字体字号格式折磨出来的工具。用 LLM 把自然语言排版指令或学校规范，自动套到 Word 文档上，重点解决了中西文字体分离这个 Word 老大难。

这个工具的起点很真实：写本科毕业论文时，我花了大量时间手动调字体、字号、标题层级这些基础格式。当时 Manus 这类 AI Agent 正在火，但实测发现它们满足不了高校对论文格式的硬性要求。所以我没指望通用 Agent，自己做了个轻量工具，只解决”能用代码确定性实现”的那部分——格式这种东西，要的是精确，不是模型即兴发挥。

把纯文本草稿，按自然语言指令（比如”正文用楷体，英文用 Arial”）或预设的学校规范，自动转成格式完整的 Word 文档。

为什么拆成两个模型

排版这件事其实是两个不同的任务，我用两个模型分管，各司其职：

  • C-Model（配置引擎）：负责解析排版规则。从自然语言指令里抽出字体、字号、行距、对齐这些参数。难点是口语化——用户说”楷体""小四”，得映射成系统认识的 KaiTi 和 12.0pt。这部分内置了一个字体映射系统专门做这种翻译。
  • B-Model（分块引擎）：负责内容结构化。识别正文里的标题、正文段、图片说明，做结构切分，同时处理 LLM 输出常见的 JSON 格式容错和符号转义。

把”理解规则”和”理解内容结构”分开，每个模型的任务更单一、prompt 更聚焦，也更好调。

真正的硬骨头：中西文字体分离

普通 Word 库设字体时，中文和英文是绑在一起的——你设了”宋体”，英文数字也跟着变宋体，很难看。但学术论文恰恰要求”中文宋体、英文 Times New Roman”这种分离设置。

我的做法是基于 python-docx 深度定制渲染器，直接操作 Word 底层 XML 的 w:eastAsia 属性，绕过普通库的限制，实现中英文双通道独立指定。还做了智能回退：如果用户没指定英文字体，系统按中文字体的类型自动配——黑体（无衬线）配 Arial，宋体（衬线）配 Times。这样即使用户只说了一半，出来的搭配也是对的。

直接操作底层 XML 还有个好处：格式在任何设备上打开都标准统一，不会因为 Word 版本不同而跑样。

配置驱动，不改代码就能扩展

字体映射表（font_config.json）和代码逻辑是分离的。想支持一个新字体（比如”方正小篆”），改配置文件就行，不用动代码；系统启动时动态加载最新配置。这让工具的可扩展性很强——把”会变的部分”外置成配置，是我做小工具时习惯的做法。

技术栈

FastAPI 提供后端 RESTful 接口，Streamlit 做前端（上传文件、输指令、实时反馈），LangChain 管 prompt 模板和链，智谱 GLM-4 负责语义理解和 JSON 格式化输出，Pydantic 做排版参数的类型校验，python-docx 负责最终生成 .docx。

小结

这是个典型的”自己的痛点驱动”的项目。它没想做成万能的，只啃下了格式里能用代码确定性解决的那块——尤其是中西文字体分离这个 Word 老大难。比起让通用 Agent 去赌，一个范围清晰、行为确定的小工具，反而更解决问题。
