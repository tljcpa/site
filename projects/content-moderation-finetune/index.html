内容合规质检模型微调

个人

把 Qwen2.5-14B 微调成一个按平台规则做内容合规质检的模型，覆盖四个平台。真正的收获不在跑通训练，而在于看穿漂亮指标背后的截断、loss mask 和数据结构问题。

时间
2026-04 ~ 2026-05
角色
数据工程 / 训练 / 评估 / 问题定位
成果
从 loss=0.0137 一路查到生产库里的数据结构缺陷；exp_006 重训把假指标变回真指标
  • Qwen2.5-14B
  • LoRA
  • DeepSpeed ZeRO-2
  • Milvus

把一个 14B 基座模型微调成内容合规质检工具：给一段文案加一批平台规则，判断违没违规、违反哪条。基座 Qwen2.5-14B-Instruct，LoRA 微调（rank=64, alpha=128, target=all），cosine 调度 lr=5e-5，bf16 + gradient checkpointing，单卡 RTX 4090 上用 DeepSpeed ZeRO-2 训练。

跑通训练只是起点。这个项目真正花时间、也真正长本事的部分，是定位一连串看起来正常、实际在骗人的指标——完整的技术调查我写成了一篇文章：微调里那些会骗你的数字。这里讲项目层面的系统设计和决策。

系统怎么设计的

覆盖四个平台：小红书、抖音、视频号、公众号。核心架构是按规则类别（category）逐一推理，再 union 结果，而不是把一个平台所有规则塞进一个 prompt 一次判完。

这个选择有讲究。一条文案常常同时踩多条规则，“多次短推理 + 合并”既能让每次 prompt 短、模型注意力集中，又能自然覆盖多处违规——某条违规在它所属类别的那次推理里被抓到，最后所有类别的结果合并。这比”一次长推理判所有规则”更稳，也更省上下文。

数据侧是一条完整的流水线：原始规则与样本 → 归一化 → 按平台/类别组织 → 生成训练样本（system prompt 装规则、user 给文案、assistant 输出 JSON 判定）。

exp_005 → exp_006：一次”把假指标变真”的重训

第一版 exp_005 训练 loss 收敛到 0.0137，全量准确率 93.6%，数字漂亮得反常。深挖后发现两个都是假象，根子是 cutoff_len=1024 太短：

  • 公众号样本的 system prompt 长达 ~9663–11505 tokens，被截断在规则中间，助手输出落在 1024 之外、不参与 loss 计算，把平均 loss 人为算低到 0.0137；
  • 评估用 padding_side="left" + 去头保尾，尾部 JSON 指令保住、头部规则被砍，模型几乎没看规则就给出了 93.6%。

确认问题不在代码、而在数据结构后（公众号 53 条规则在生产 Milvus 库里只挂一个 category，我连进去核对过），做了 exp_006 重训：cutoff 提到 4096、规则加主题分组、重新生成数据。

关键决策是认清 4096 的边界：它装得下小红书/抖音/视频号最大的 category（视频号最大一组 3438 tokens 刚好够），但装不下公众号的 11505 tokens。所以 exp_006 对公众号只是”梯度从≈0 变有效”（从只看到约 3 条规则变成约 14 条），推理仍要分 3–4 批跑完合并——是部分修复，不是一次到位。

重训后 train_loss 是 0.245（不是变差，是所有 token 都参与计算后的真实值，和 0.0137 不可比）。

留下的判断习惯

这个项目给我最大的收获不是某个超参，而是一套不信任漂亮数字的反射：

  • 异常好的 loss，先查是不是截断/loss mask 把它算小了；
  • 高准确率，先核对模型实际看到的输入完不完整（padding 方向、截断方向）；
  • 线上差但指标好，先比训练分布和推理分布的差异；
  • 指标异常时，顺着输入一路查到数据源——这次的终点是生产库里”公众号只有 1 个 category”，一个数据结构问题，纯靠训练补不回来。

模型训练里最危险的不是会报错的地方，而是这些看起来一切正常、却在悄悄骗你的指标。
