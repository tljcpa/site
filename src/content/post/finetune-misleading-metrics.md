先交代背景。这是一个内容合规质检模型：给定一段文案和一批平台规则，判断它违没违规、违反了哪条。基座是 Qwen2.5-14B-Instruct，LoRA 微调（rank=64, alpha=128, target=all），覆盖四个平台——小红书、抖音、视频号、公众号。

训练完，两个指标好得让人想直接收工：train_loss **0.0137**，全量准确率 **93.6%**。

这篇就是讲，这两个数字怎么一步步被我自己拆穿，以及它们最后一起指向的那个真问题——它不在模型里，在数据结构里。

## 先理解架构：为什么是"按类别逐一推理"

这点不讲清楚，后面的 bug 没法理解。

一条文案可能同时踩多条规则。直觉做法是把一个平台的所有规则塞进一个 prompt，让模型一次性判完。但我们的架构不是这样——它是**按规则类别（category）逐一推理，再把结果 union**：

```
文案："买了就送礼，AI画像帮你定制专属方案"
  ↓
对「营销类别」(8条规则)推理   → 违规1：夸大效果
对「AI内容类别」(5条规则)推理 → 违规2：AI功能误导
对「其他类别」...             → 无违规
  ↓
union → violations: [违规1, 违规2]
```

这样设计有两个好处：每次 prompt 短（只装一个类别的几条规则），模型注意力集中；一条文案的多处违规，在不同类别的推理里分别被抓到，最后合并。**多处违规是靠"多次短推理 + 合并"覆盖的，不是靠"一次长推理"。** 记住这句，它是后面所有问题的钥匙。

## 症状：公众号在胡说

四个平台里，小红书、抖音、视频号都正常，唯独公众号的输出是这样的乱码——模型不输出 JSON 判定结果，而是在**续写规则文本**（`支付费用后可获得投票资格...`）。

模型在续写输入，说明它根本没读到那条"请输出 JSON"的指令。为什么？

## 第一层：prompt 9663 tokens，训练 cutoff 1024

去量了一下：公众号每条样本的 system prompt 长达 **9663 tokens**，而训练时 `cutoff_len` 设的是 **1024**。

prompt 被从中间截断，模型在训练时压根没看到末尾的 JSON 指令，自然学不会在那个位置输出 JSON。推理时它看到一段被截断、停在规则中间的输入，就顺着续写下去了。

到这一步，"乱码"解释通了。但真正有意思的是它牵出的两个假象。

## 假象一：loss=0.0137 是怎么被算低的

LoRA 微调收敛到 0.0137 是反常的——这种量级要么过拟合，要么哪里不对。真相是：它被**截断 + loss mask**联合算低了。

训练时只在"助手输出"那部分 token 上计算损失（loss mask）。而公众号、视频号这些长样本，助手输出常常落在 1024 之后，**被截断的部分根本不参与损失计算**。难样本的难 token 大量缺席，平均下来 loss 自然低。

0.0137 不代表模型学得好，它代表大量该算的 token 没被算进去。

后来把 cutoff 放开到 4096 重训（exp_006），同样的数据、同样的模型，train_loss 变成 **0.245**。所有 token 都参与计算后，这才是真实水平。

> **结论：loss 的绝对值取决于你在多少 token、哪些 token 上计算它。截断不同、loss mask 不同，loss 就不可比。** exp_005 的 0.0137 和 exp_006 的 0.245 之间没有"变差"，只是前者算的是一个缩了水的子集。

这里还有个容易忽略的细节：对公众号样本，cutoff=1024 时模型只看得到大约**3 条规则**，对应的梯度信号接近于零——也就是说，公众号这部分数据在训练里近乎"空转"，既没贡献有效 loss，也没贡献有效梯度。

## 假象二：93.6% 准确率是在"没看规则"的情况下蒙的

奇怪的是，全量评估时公众号还有 95.6% 的 JSON 有效率、93.6% 的准确率。一个连规则都没读到的模型，怎么还能答对？

去拆评估的实际输入，问题浮出来。批量评估用了 `padding_side="left"`，超长时的截断方向是**去头保尾**：

- **尾部保住了**——那段"请输出 JSON"的指令在结尾，所以模型还能吐出格式合法的 JSON（95.6% JSON 有效率的由来）；
- **头部被砍了**——而**规则正文都在前面**，被截掉了。模型实际只看到最后 1024 个 token。

也就是说，这 93.6% 是模型在**几乎没看到规则**的情况下做出的判断。它大概率学到了一个先验——多数文案是合规的——靠这个先验蒙，也能到九成。这个准确率衡量的不是"按规则判断的能力"，而是"没有规则时的基线猜测"。

> **一个评估数字，如果你不去核对模型到底看到了什么输入，就可能把流程里的截断 bug，误读成模型的能力。**

## 一个好问题：数据没超 1024，为什么也不行

调查到这里，被问了一句很关键的话："我们的数据集似乎并没有超过 1024，为什么也不行？"

这正是最容易混淆的点。超长的不是**数据样本**（那段待判定的文案很短），超长的是**system prompt**——也就是塞进去的那一大堆规则。一条公众号样本 = 短文案 + 53 条规则的长 prompt，长度由规则贡献，不由数据贡献。所以"数据没超 1024"和"样本被截断"两件事并不矛盾。

## 根因：这是数据结构问题，不是代码 bug

为什么偏偏是公众号？我把四个平台的规则结构摆在一起：

| 平台 | 规则数 | category 数 | 每批最多规则 | prompt 量级 | 状态 |
|------|-------|------------|------------|-----------|------|
| 小红书 | 49 | 14 | 6 条 | 短 | 正常 |
| 抖音 | 107 | 21 | 10 条 | 短 | 正常 |
| 视频号 | 135 | 11 | 18 条 | 最大一组 3438 tokens | 基本正常 |
| **公众号** | **53** | **1** | **53 条** | **11505 tokens** | 根本问题 |

差别一目了然：别的平台规则分成了十几个 category，每次推理只装一个 category 的几条规则，prompt 很短；**公众号的 53 条规则在数据源头就只挂在一个 category 下**，没有子分组，于是"按类别推理"退化成"把 53 条全塞进一个 prompt"，直接 11505 tokens。

为了确认这不是本地代码的问题，我连进了生产环境的 Milvus 向量库去核对——公众号在生产库里同样是 53 条全在一个 category 下，和本地一致。**所以这是数据从源头就缺了一层分类结构，不是推理代码写错了。** 其他三个平台的架构完全没问题。

## 修复，以及它的诚实边界

两条路：

- **路线 A（不重训）**：手动把 53 条规则按主题分成 6–8 组，推理时按组跑、合并。能解决，但现有模型没见过这种分组格式。
- **路线 B（重训 exp_006）**：给公众号规则加主题分组、把 cutoff 从 1024 提到 4096、重新生成训练数据、重训。

选了 B，但要说清楚它的边界——**4096 也装不下公众号的 11505 tokens**。提到 4096 的真实意义不是"一次装下所有规则"，而是把公众号从"截断到 1024、只看到约 3 条规则、梯度≈0"，变成"截断到 4096、能看到约 14 条规则、有真实梯度"。公众号推理时仍然要分 3–4 批（每批约 15 条）跑完再合并。

所以 exp_006 实际解决了三件事，且程度不同：

1. 抖音/视频号的训练截断——**完整修复**（4096 能装下它们最大的 category，视频号最大一组 3438 tokens 刚好够）；
2. 公众号训练梯度从零变有效——**部分修复**；
3. 公众号推理的分批策略——训练之外还得单独实现。

代价也真实：4096 cutoff 下吞吐从 exp_005 的 15 samples/s 掉到 2.03 samples/s，**慢约 7 倍**，整轮训练 2 小时 48 分（ZeRO-2 扛住 4096，显存 37283/49140 MiB，没 OOM）。

更重要的是那句没法绕过的话：**一个数据结构缺失的问题，没法纯靠训练补回来。** 重训能让梯度有效、让 loss 变真实，但公众号"53 条挤一个 category"这个结构问题，最终得在数据侧补上分组才算真解。

## 留下的几条习惯

- **异常好的 loss 先查 mask 和截断**。低 loss 可能只是因为算的 token 少、被截掉的 token 没进损失。不同截断/掩码下的 loss 不可直接比较。
- **高指标先核对模型的真实输入**。padding 方向、截断方向、上下文有没有被砍，决定了这个数字到底在衡量什么。`padding_side="left"` + 去头保尾，能让一个没读到规则的模型照样吐合法 JSON、照样"准确"。
- **线上差、指标好，去比训练分布和推理分布**。训练每次 1 个 category，推理一次 53 条，这种 train-serve skew 不体现在 loss 上、也不体现在同分布评估上，只在真实使用时暴露。
- **指标异常时，顺着输入一路查到数据源**。这次的终点不是某个超参，是生产库里"公众号只有 1 个 category"。很多看起来像模型的问题，根子在数据结构。

微调里最危险的从来不是会报错的地方——那些你迟早会修。危险的是这些看起来一切正常、却在悄悄骗你的数字。
